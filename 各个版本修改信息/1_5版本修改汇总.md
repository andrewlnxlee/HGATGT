  ---
  v1.5.2 → v1.5.6 结论总结

  1. 总结论

  这几轮迭代最终证明：

  - v1.5.2 的问题不是 H-GAT-GT 模型整体失效
  - 真正出问题的是 点级评估/后处理链路
  - 问题分成两个阶段：
    a. clutter 被带进点级预测链路，导致数量指标先崩
    b. H-GAT-GT 的点级 ID 生成方式不公平，导致 IDSW 极高、MOTA 被压低

  最终在 不重训 checkpoint 的前提下，通过修正点级评估链路，就把 H-GAT-GT 的点级结果从“几乎不可用”拉回到第一梯队。

  ---
  2. 各版本核心结论

  ---
  v1.5.2：问题暴露期

  现象
  - 群级结果其实很好，说明模型主干和群级 head 没崩
  - 但点级结果非常差：
    - H-GAT-GT (Ours)
        - OSPA(Total) = 11.7546
      - OSPA(Card) = 11.6408
      - Count Err = 8.0570
      - FAR = 8.0570
      - MOTA = 0.0554
  - 同时 Ours 比 Meas Ablation 仍然有更好的：
    - OSPA(Loc)
    - RMSE(Pos)

  结论
  - 这说明：
    - point head 的位置修正是有效的
    - 但点级总指标被 clutter / 数量错误 / 后处理关联 淹没了
  - 所以第一判断是对的：
    - 先修评估链路，不要先怀疑模型或急着重训

  ---
  v1.5.3：第一轮关键修复成功

  改动
  - 在 sim_env/evaluate_single.py 里把 H-GAT-GT 的点级预测主链路改成：
    - corrected_pos
    - DBSCAN
    - 只保留非噪声点
    - 再送入点级 tracker
  - 诊断行 Meas Ablation 也同步改成同样的 filtered 口径

  结果
  - H-GAT-GT (Ours)：
    - OSPA(Total): 11.7546 -> 2.5092
    - OSPA(Card): 11.6408 -> 1.1116
    - Count Err: 8.0570 -> 0.2452
    - FAR: 8.0570 -> 0.2056
    - MOTA: 0.0554 -> 0.3122

  结论
  - 第一轮修复直接证实：
    - 点级大崩的第一主因就是 clutter 被错误带进点级 tracking
  - 同时：
    - Ours 仍继续优于 Meas Ablation 的 OSPA(Loc) / RMSE(Pos)
  - 这说明：
    - point head 的增益是真实存在的
    - 不是模型没学到，而是链路把收益吃掉了

  ---
  v1.5.4：错误方向验证

  改动
  - 尝试调整点级 tracker 阈值：
    - stage1=14
    - recovery=22
    - max_age=20

  结果
  - OSPA / RMSE / Count Err / FAR 基本不变
  - 但 IDSW 明显变差：
    - 110925 -> 125473
  - MOTA 下降：
    - 0.3122 -> 0.2230

  结论
  - 这证明：
    - 当前问题的主矛盾不是点级 tracker 阈值没有调准
    - 继续在 gnn_processor_single.py 上盲调，只会恶化关联
  - 所以这一轮应判定为：
    - 无效甚至有害的尝试
    - 已回退

  ---
  v1.5.5：metrics 口径排查

  改动
  - 在 metrics.py 里给 IDSW 增加 gap tolerance 逻辑
  - 对比：
    - POINT_ID_SWITCH_GAP_TOLERANCE = 0
    - POINT_ID_SWITCH_GAP_TOLERANCE = 1

  结果
  - gap=0：
    - IDSW = 103764
    - MOTA = 0.3561
  - gap=1：
    - IDSW = 110596
    - MOTA = 0.3142

  结论
  - gap=0 更好
  - 说明在当前 detected-only GT 场景里：
    - 允许更宽的 gap 反而没有带来更合理的 MOTA
  - 最终结论：
    - POINT_ID_SWITCH_GAP_TOLERANCE = 0 保留
    - metrics 不是下一步继续优化的主方向

  ---
  v1.5.6：第二轮关键修复成功

  改动
  - 把 H-GAT-GT 的点级 ID 生成从“纯点级 tracker”改成：
    - 群级 track -> 点级 group-constrained association
  - 具体是：
    - 先对 H-GAT point head 输出点做聚类
    - 用 GNNPostProcessor 跟踪群
    - 再把群级 stable ID 投影回点
    - 最后用 GroupConstrainedPointAssociator 做点级 ID
  - Meas Ablation 也同步使用同一条身份生成链路，保证公平

  结果
  - H-GAT-GT (Ours)：
    - IDSW: 103764 -> 44064
    - MOTA: 0.3561 -> 0.7222
  - 同时几乎没有破坏之前已经修好的结果：
    - OSPA(Total) = 2.5092
    - OSPA(Loc) = 1.6671
    - OSPA(Card) = 1.1116
    - RMSE(Pos) = 1.7551
    - Count Err = 0.2452
    - FAR = 0.2056

  结论
  - 这轮直接命中了第二主因：
    - H-GAT-GT 点级 ID 生成结构不公平
  - baseline 能有较低 IDSW，不是因为它们定位更强，而是因为它们本来就在用：
    - group-constrained point association
  - H-GAT-GT 一旦复用同类结构，MOTA 立刻上来了

  ---
  3. 最终问题定位

  经过 v1.5.2 → v1.5.6，可以把问题归纳成两层：

  第一层问题：数量错误

  - 根因：预测链路把 clutter 一起送进点级 tracking
  - 解决版本：v1.5.3
  - 主要影响指标：
    - OSPA(Card)
    - Count Err
    - FAR

  第二层问题：身份连续性错误

  - 根因：H-GAT-GT 点级 ID 生成方式是纯点级全局关联，没有复用群级稳定信息
  - 解决版本：v1.5.6
  - 主要影响指标：
    - IDSW
    - MOTA

  4. 最终保留的有效设置

  当前应保留的正式实现包括：

  点级预测链路

  5. 当前最终结论

  截至 v1.5.6，可以给出比较明确的结论：

  ▎ H-GAT-GT 的 point head 本身是有效的，v1.5.2 的差结果主要不是模型没学到，而是点级评估链路先后放大了两个问题：
  ▎ 1. clutter 被错误带入点级 tracking，导致 cardinality 崩溃；
  ▎ 2. H-GAT-GT 的点级 ID 生成没有复用现成的群级时序稳定信息，导致 IDSW 极高。
  ▎ 依次修正这两个问题后，在不重训的情况下，H-GAT-GT 点级结果已从 MOTA ≈ 0.055 提升到 MOTA ≈ 0.722，同时保持了优于 raw measurements      
  的定位优势。

  组约束关联，最终在不重训的前提下把点级 MOTA 从 0.055 提升到 0.722。
